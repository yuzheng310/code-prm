## ⚠️ 2026-05-30 独立核对结果(覆盖上面的 GO 结论)

把 `data/code-trajectory-2.4k/{train,val,test}.jsonl` 拷到本地做了独立核对。**数据完整性 OK**:文件大小正常(train 9.6MB、val/test 各 ~1.1MB)、行数与 pass rate 和文档一致(474/59/59;29.96% / 25.42% / 33.90%)、148 task × 4 rollout 一条不缺。但发现两个**阻断 Phase 2/3 的问题**,外加标签退化被坐实。

### 阻断项 ① 任务级泄漏 = 100%

切分用的是 trajectory 级随机洗牌(`scripts/30_assemble_dataset.py` 的 `split()`,无 task 分组),同一 task 的 4 个 rollout 被打散到不同 split:

| 指标 | 实测 |
|---|---:|
| train 覆盖的 task | 148 / 148 |
| test 的 task 同时也在 train | 47 / 47 |
| val 的 task 同时也在 train | 54 / 54 |
| test 中干净(train 没见过)的 task | 0 |
| 出现在 >1 个 split 的 task | 86 / 148 |

→ 每条 val/test 轨迹所属的 task,其它 rollout 都在 train,held-out 指标被系统性高估。

### 阻断项 ② held-out Best-of-N 无法构造

- mixed-outcome task 共 28 个,其中 17 个的 rollout 跨多个 split。
- **4 个 rollout 全落在 test 内的 mixed task = 0** → test split 内凑不出任何干净的 per-task BoN 候选集,Phase 3 核心实验当前做不了。

### ③ 标签退化(坐实)

- 2540 个 step **全部是 tool step**(无纯 thought step)→ masked-MSE 的 mask 形同虚设。
- step_label 分布 `{0.0: 2153, 0.25: 137, 0.5: 65, 0.75: 55, 1.0: 130}`:**84.8% 为 0**,非零仅 387(15.2%),≥0.5 仅 250(9.8%)。
- 恒输出 0 的平凡预测器 masked-MSE ≈ 0.073(RMSE ≈ 0.27),模型易退化成 ORM。

### 修正后的结论

- 上文 `质量报告 decision: GO` 只在「raw/labeled 结构完整 + 流程可复现」层面成立。
- 旧 `data/code-trajectory-2.4k/` **不能作为 Phase 2/3 输入**。
- `scripts/30_assemble_dataset.py` 已修复为 task-grouped split；重新 assembly 后的新目录 `data/code-trajectory-2.4k-tasksplit/` 才是可信输入。

### 已实施的 split 修复

1. `split_by_task()` 按 **`task_id` 整组切分**:一个 task 的 4 个 rollout 全部进入同一 split。
2. 按 pass-count 桶 `0/4,1/4,2/4,3/4,4/4` 分层,减少 split 间 pass-rate / 任务组成漂移。
3. 默认 mixed-outcome task 分配为 `train/val/test = 18/4/6`,保留 train ranking 信号,同时让 test 有完整 held-out BoN 候选集。
4. 新增 hard checks:无 task overlap、每 task rollout ids `[0,1,2,3]` 完整、592 条守恒无丢无重、val/test mixed 数达标。
5. `split_manifest.json` 已记录 seed、策略、输入、split 统计、pass-count histogram、mixed task ids、以及上述 hard-check 结果。
6. Phase 2 评测须带 baseline(零预测器 + outcome-only ORM),并在「673 真信号步」子集上单独报指标。
