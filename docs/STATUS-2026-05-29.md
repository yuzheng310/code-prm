## 今天完成的关键修复

### 1. BigCodeBench grader harness 修复

文件:`src/collector/trajectory_logger.ts`

当前 harness 生成逻辑已改为:

```python
from task import *
from task import <entry_point> as task_func
<dataset test code>

if __name__ == "__main__":
    unittest.main()
```

修复的问题:
- 原先 `python <test_file>` 可能 0 tests ran 但 exit 0，产生假阳性。
- BigCodeBench 部分 test 会引用 `task.py` 顶层 import 的符号，例如 `pd`、`WordCloud`、`RandomForestClassifier`；只 import entry point 会导致 grader namespace `NameError` 假失败。

### 2. per-rollout cwd 隔离

文件:`src/eval/swebench_runner.py`

BigCodeBench 每个 task/rollout 现在使用独立 cwd:

```text
data/raw/bigcodebench-hard/_workdirs/<task>__rollout_<k>__<run_id>/
```

这避免 `--concurrency 4` 时共享 `task.py`、grader 文件、`__pycache__`、模板文件等。

### 3. BigCodeBench prompt 修复

prompt 现在包含:
- `instruct_prompt` / `prompt`
- `code_prompt`
- 精确 `entry_point`
- 明确要求在当前 cwd 的 `task.py` 中实现指定入口

### 4. raw / labeled / quality 审核脚本

新增/增强:

| 脚本 | 用途 |
|---|---|
| `scripts/08_audit_pilot.py` | raw trajectory 结构与 grader 可信度审核；现在支持 full 4-rollout 审核 |
| `scripts/09_audit_labeled_pilot.py` | labeled pilot 审核；检查 label_method、step_label 覆盖率、outcome=0 标签一致性 |
| `scripts/12_report_phase1_quality.py` | Phase 1 质量报告；输出 `summary.json`、`report.md`、SVG 图表 |

## 当前风险 / 注意事项

1. **不要重启第二个全量采集进程**  
   会产生重复支出和可能混写。若怀疑卡住，先看进程、日志、jsonl 行数。

2. **不要在 raw 审核前跑 labeling**  
   labeling 成本高；raw outcome 一旦有系统性问题，labeling 会把错误数据放大。

3. **`08_audit_pilot.py` 名字仍是 pilot，但现在支持 full raw 审核**  
   全量必须带:
   ```bash
   --expected-count 592 --expected-rollouts-per-task 4
   ```

4. **质量判断不等于格式判断**  
   hard audit 只证明数据结构和 grader 没坏；`12_report_phase1_quality.py` 才判断 pass-rate、mixed tasks、tool-step 分布、label 分布等是否支撑后续 PRM。

5. **BigCodeBench-Hard v0.1.4 只有 148 tasks**  
   当前 4 rollouts 只能得到 592 trajectories，不是旧叙事里的 1200/2400。若仍追求更大规模，后续需要 8 rollouts 或补充任务源。
