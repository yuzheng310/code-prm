第 6 篇 · 深度与求职

深度、批判与求职

前五篇让你「懂」。这一篇让你「能在面试里赢」。把项目里那些被简化掉的、有风险的、可质疑的点全部摊开,逐个给出「面试官会怎么问 / 你怎么答 / 怎么补成真深度」。一个能批判自己项目的人,远比一个只会夸自己项目的人可信。

6.1残酷的前提:Phase 1 不补你的短板

你的目标是字节 Seed 的 Reward / Agent 算法岗,自评短板是没有 ML 训练经验。要清醒地认识到:Phase 1 一行训练代码都没有。 不管采集/labeling 管道多漂亮,它在结构上无法弥补「没训过模型」这个短板。真正补短板的是 Phase 2(你真的训了一个 PRM) 和 Phase 3(它真的有用)。

但 Phase 1 对 Reward 岗有一个独特价值

Reward modeling 这个方向,数据质量和标签噪声就是一切。同一个 Phase 1,当「管道」做一文不值,当「奖励数据质量研究」做就正中要害。区别只在于:你有没有量化、验证、批判你自己的标签。下面几节就是教你把 Phase 1 从前者升级成后者。

6.2批判点一:outcome_zero 简化注入了标签噪声

回忆第 2/4 篇:失败轨迹(outcome=0)的每个 tool step 都被直接标 0。问题显而易见:

一条最终失败的轨迹,前面 8 步可能完全正确,只是第 9 步崩了。把前 8 步也标 0,等于告诉模型「这些好步是坏的」——这是系统性的标签噪声。
面试官会问 / 你怎么答

问:「失败轨迹里正确的中间步你也标 0,不怕训出错误信号吗?」

答(诚实 + 有方案):「会引入噪声,这是 Math-Shepherd outcome-only 简化的已知弱点。我之所以先这么做,是因为它把失败路径的 API 成本降到 0;但我清楚它的代价,所以(a)我用 label_method 显式标记了这批是简化标签,可在 ablation 里单独剔除;(b)Phase 2 的对比实验之一就是『失败轨迹也跑 judge』vs『直接标 0』对 Best-of-N 的影响。」

6.3批判点二:LLM-judge 没被验证过(最该补的实验)

这是对 Reward 方向最致命的缺口:你用 judge 的 PASS 比例当 step 价值,但你从没验证过 judge 估的准不准。「你怎么知道你的奖励信号是对的」是这个方向的第一问。

补法:surrogate 验证实验(便宜、出硬数字)

挑一个小子集(比如 20 条轨迹、~150 个 step),对这些 step 真的跑一次蒙特卡洛 rollout(从该步重启 agent + 跑真实测试 N 次),得到「真值」step 价值。然后量两者的一致性:

  • Pearson / Spearman 相关系数:judge 分数 vs 真 rollout 分数,$\rho$ 多少?
  • AUC:用 judge 分数预测「真 rollout ≥ 0.5」的能力。
  • 校准曲线:judge 说 0.6 的那些步,真实成功率是不是真在 0.6 附近?

这给你一句硬话:「我用 N=8 的真 MC rollout 在 20 条轨迹上验证了 LLM-judge 代理,二者 Spearman ρ=0.X,校准在 [0.3,0.7] 区间良好、两端偏乐观。」 这一句直接把「偷懒的 surrogate」变成「被验证过、知道边界的 surrogate」——这是研究员和工程师的分水岭。

6.4批判点三:reward hacking

名词解释:reward hacking(奖励钻空子)

策略模型学会「讨好奖励模型」而非真正解决问题。比如 PRM 偏爱「先读很多文件再动手」,策略就学会狂读文件刷高分,哪怕没用。这是 R1 放弃 PRM 的理由之一,也是 Reward 方向的经典议题。

你怎么答

「代码场景对 reward hacking 有天然防线:最终还有真实测试这个不可造假的 outcome 兜底,PRM 只用在 Best-of-N 的候选重排,不直接做长程 RL 优化,所以被 hack 的暴露面小。如果未来上 RL,我会监控『PRM 分数涨但真实 pass 率不涨』这个背离信号——那就是 hacking 的指纹。」

6.5批判点四:数据太小(148 个任务)

BigCodeBench-Hard v0.1.4 只有 148 个任务。要凑到约 1200 条轨迹得每题 8 个 rollout。这对训一个稳健的 PRM 是偏小的。但你手里有张好牌:data/prm800k/(80 万条数学步骤标签)。

把数据荒变成一个研究问题:跨域迁移

提出问题:「过程奖励能跨域迁移吗?——在数学(PRM800K)上训的 PRM,在代码 Agent 上还管用吗?」 这一个 ablation 同时解决三件事:① 绕开 148 的数据荒(用海量数学数据预热);② 提供一个有研究味道、且当前文献尚无定论的实验;③ 给面试官又一个钩子。可设计成:纯代码训练 vs 数学预热+代码微调 vs 纯数学,对比三者的 Best-of-N。

6.6元问题:PRM 到底值不值得做?

最高级的批判是质疑项目本身的前提。一个诚实的 Phase 3 应该包含一个 baseline:纯 ORM / 纯 RLVR(只用测试结果,不要过程标签)。如果你的 PRM 在 Best-of-N 上打不过这个简单 baseline,那就是个有价值的负面结果——它印证了 R1 的判断,而你能讲清「在我的设定下,过程监督的收益没有覆盖它的标签噪声成本」。

面试视角:负面结果也是结果

很多人怕做出「PRM 没赢」的结果。但对 Reward 方向,一个带着清醒分析的负面结论,比一个「不知道为什么赢了」的正面结论更能体现科学素养。关键是你能不能解释清楚为什么。永远准备好对比 baseline,永远准备好解释结果——无论正负。

6.7把项目讲成一个故事(电梯版)

30 秒自述模板

「我在做一个面向代码 Agent 的过程奖励模型。出发点是:DeepSeek-R1 放弃 PRM 是因为纯推理的『步』太模糊、难标、易 hack——但代码 Agent 的中间状态是可验证的(能跑吗、测试过吗),所以过程监督在这里反而成立。我用一个真实的 TS 编码 Agent 采集多轮轨迹,outcome 用 BigCodeBench 官方测试判定,step 标签用一个 GenRM 风格的 LLM 判官按 Math-Shepherd 的 soft-estimation 方式生成,并对标签来源做了可审计的标注和 6 道质量闸门。我还用真 MC rollout 在子集上验证了这个代理估计器的校准。最终我训了一个 Qwen-Coder-1.5B + LoRA 的 scalar-head PRM,在 Best-of-N 上把 pass@1 提升了 X%,并和纯 ORM baseline 做了对比。」

注意这段话里每个加粗词,都对应前面某一篇的一个知识点。它之所以有说服力,是因为它既讲了你做了什么,也讲了你为什么这么选、代价是什么、怎么验证的——这正是 Reward 方向想要的「研究员心态」。

6.8优先级建议

  1. 先修 grader + cwd 隔离(第 4 篇的两个坑)——outcome 不可信,后面全白搭。
  2. 尽快推进 Phase 2/3,拿到那个 Best-of-N 数字——这是补「没训过模型」短板的唯一途径,也是项目价值的落点。
  3. 做 §6.3 的 surrogate 验证实验——成本低、对 Reward 岗杀伤力最大。
  4. 把 §6.5 的数学→代码迁移作为 ablation——同时解决数据荒和「研究味道」。
  5. 不要再在 Phase 1 管道的工程精致度上加投入了。够用即可。

  • Zhang et al. (2024). Generative Verifiers: Reward Modeling as Next-Token Prediction. — GenRM,你 LLM-judge 的趋势依据。

  • Lambert et al. (2024). Tülu 3. — RLVR 的系统化实践。

你现在应该能:从零向别人解释这个领域;对着代码说出每一行的来历;在面试里回应 R1 之问、讲清 GenRM 定位、承认并量化自己的简化代价。剩下的就是把 Phase 2/3 跑出来,让那个 Best-of-N 数字落地。回到 总览 或任选一篇重读深水区。
