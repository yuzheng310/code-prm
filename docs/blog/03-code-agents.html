第 3 篇 · 核心论点

把 PRM 搬到代码 Agent

PRM 是在「数学解题」上证明的。把它搬到「多轮代码 Agent」上,不是简单换个数据集——「一步」的定义变了,而且要直面一个尖锐的质疑:DeepSeek-R1 明确说过「我们放弃了 PRM」。这一篇讲清楚为什么代码 Agent 反而是过程监督更合适的场景,以及这背后那个能撑起半场面试的论点。

3.1代码 Agent 的轨迹长什么样

数学题的「过程」是一串文字推理(CoT,思维链),一步 = 一行推导。但一个代码 Agent 解决任务时,产生的是一条多轮工具调用轨迹 (multi-turn tool-use trajectory):

任务:实现 task.py 里的 parse_config 函数,让单元测试通过

Step 0  thought: 我先看看测试想要什么
        tool: read_file(tests.py)        → 读到测试代码
Step 1  thought: 需要处理 YAML,先看现有结构
        tool: read_file(task.py)         → 看到空函数
Step 2  tool: write_file(task.py, "...") → 写下第一版实现
Step 3  tool: run_bash("python -m pytest")→ 3 个测试挂了 2 个
Step 4  thought: 边界情况没处理
        tool: edit_file(task.py, "...")  → 修补
Step 5  tool: run_bash("python -m pytest")→ 全过 ✓
        → outcome = 1
一条代码 Agent 轨迹。每个 Step 是一次「想法 + 工具调用 + 工具返回」。这正是 trajectory_schema.py 里的 Step 结构。
名词解释:在代码 Agent 里,「一步」= 一次工具调用

这是一个设计选择,不是天经地义的。你也可以定义「一步 = 一个 token」或「一步 = 一整轮对话」。本项目定义为「一次 tool call」(read_file / write_file / run_bash …)。纯思考(没有 tool 的 step)默认不打标签。粒度的选择会影响信号质量——这本身就是个可讨论的研究点。

3.2关键论点:代码 Agent 的中间状态是可验证的

这是整个项目最重要的一句话,请慢读。

回忆第 2 篇:PRM 的标签来自「从这一步出发最终能成功的概率」。这要求你能判断成功。在数学里,只有走到最后才能对答案——中间步骤没有客观的「对错」可言(一步推导是否「好」本身就模糊)。

但代码 Agent 不一样。它的中间状态有大量客观、可机器验证的信号:

  • 这一步写的代码能不能跑(语法/import 错误立刻暴露)?
  • 跑测试过了几个(从挂 2 个到挂 0 个,是看得见的进展)?
  • 文件改对地方了吗(diff 可检查)?
核心论点(请背下来)

数学推理的「步」缺乏客观的中间正确性,所以过程监督在那里又贵又噪;而代码 Agent 的中间状态天然可验证(能跑吗、测试过吗、改对吗),所以过程监督在代码场景里更可行、也更值得做。

3.3必须正面回答:「DeepSeek-R1 不是放弃 PRM 了吗?」

面试视角:这是必考题

任何懂 RL 的面试官看到你做 code PRM,第一反应就是:「DeepSeek-R1 的技术报告里明说试过 PRM 然后放弃了,你为什么还做?」答不上来 = 当场出局。答得好 = 直接证明你读懂了领域。

先把 R1 的理由摆出来(它说得没错):DeepSeek-R1 报告中放弃 PRM 的三条理由是——

  1. 难以定义细粒度的「步」:在一般推理里,一步到底是什么,边界模糊。
  2. 难以判断中间步的对错:自动标注不准,人工标注不可规模化。
  3. 容易 reward hacking:模型会学会「讨好」PRM 而非真正解决问题;且 PRM 要反复重训,工程开销大。

然后给出你的反驳——注意,不是说 R1 错了,而是说它的结论限定在「纯推理」场景:

你的反驳

R1 的三条理由,在代码 Agent 上都被显著削弱:

① 步的定义在 agent 里是天然的——一次工具调用就是一步,边界清晰。
② 中间步的对错有客观抓手——代码能不能跑、测试过几个,不全靠模型瞎猜。
③ reward hacking 的空间更小——因为最终还有真实测试这个不可造假的 outcome 兜底。
所以:R1 在纯推理放弃 PRM 是合理的;但这恰恰反过来说明,中间状态可验证的代码 Agent,正是过程监督值得重新尝试的地方。

3.4第二个趋势:用 LLM 当判官(GenRM)

第 2 篇说本项目把「真 rollout」换成了「让一个 LLM 判官预测会不会成功」。这听起来像偷懒,但其实踩在一条正经的学术趋势上,叫 GenRM(Generative Reward Model / Generative Verifier,Zhang et al., 2024)。

名词解释:GenRM(生成式奖励模型)

传统 RM 是「输入文本 → 输出一个分数」(判别式)。GenRM 反其道:把「判断好坏」当成一个生成任务——让一个 LLM 直接说出「这个对/错」「Yes/No」,再用它生成 "Yes" 的概率当分数。好处:能复用 LLM 的推理能力、能写出判断理由(chain-of-thought 式验证)、不需要专门训一个判别头。

看你项目的 judge prompt(_build_continuation_prompt):它把「问题 + 到目前为止的轨迹」喂给 LLM,要求它回一行 OUTCOME: PASS 或 OUTCOME: FAIL,可选附理由。然后跑 K 次,数 PASS 的比例。这就是一个 GenRM 风格的过程价值估计器。

面试视角:把「偷懒」讲成「趋势」

不要说「我用 LLM judge 是因为真 rollout 太麻烦」。要说:「我采用了一个 GenRM 风格的生成式判官作为过程价值的代理估计器(surrogate),用 K 次采样的 PASS 比例作为 soft label,这与 Zhang et al. 的 Generative Verifier 路线一致;它牺牲了一定的标签保真度,换取了数量级的成本下降。」 同一件事,后一种讲法立刻显出你知道自己在做什么。

3.5但要诚实:LLM-judge ≠ 真 rollout

必须知道代价,否则面试官追问会露馅。step_labeler.py 的文档注释自己写得很清楚:

THIS IS NOT MONTE-CARLO ROLLOUT. … This module's surrogate is much cheaper but is a WEAK SUPERVISION signal biased by the judge's calibration.
真 MC rollout本项目 LLM-judge 代理
怎么估成功率真的从该步重跑 agent + 跑真实测试 K 次让 judge 预测会不会成功 K 次
需要恢复状态(repo 快照)+ 沙箱重执行只要把前缀文本喂给 API
成本极高(每步要跑 agent+测试)低(几次 API 调用)
偏差来源采样方差(可控)judge 自身的校准偏差(不可控,需验证)

schema 里特意保留了 repo / base_commit / final_diff 这些「replay 字段」,就是为了 Phase 2 有能力升级到真 MC rollout——label_method 会从 "llm_judge" 改成 "mc_rollout",字段语义不变。这种「为未来留接口 + 诚实标注当前方法」的设计,本身是工程成熟度的体现。

3.6本篇小结

  • 代码 Agent 的轨迹是多轮工具调用;本项目定义「一步 = 一次 tool call」。
  • 核心论点:代码 Agent 的中间状态可验证(能跑/测试过/改对),所以过程监督比在纯推理上更可行。
  • 必背:R1 放弃 PRM 是针对纯推理(步模糊、难标、易 hack);这三条在代码 Agent 上都被削弱。
  • 用 LLM 当判官 = GenRM 趋势,把判断当生成任务;本项目的 judge 就是一个 GenRM 式过程价值估计器。
  • 但要诚实:LLM-judge 是代理(surrogate),有 judge 校准偏差,不等于真 rollout;schema 已留好升级接口。
