奖励模型基础:从 RLHF 说起

在谈「过程奖励」之前,得先回答一个更朴素的问题:大模型明明已经会说话了,为什么还需要一个专门的模型来告诉它「什么是好答案」?这一篇把奖励模型放进 RLHF 的全景里,讲清它的位置、训练方式,以及代码任务为什么是个特殊的「免训练奖励」乐土。

1.1预训练模型只会「接话」,不懂「好坏」

一个刚预训练完的大语言模型(LLM),本质上只学会了一件事:给定前面的一串 token,预测下一个 token 最可能是什么。它读了整个互联网,所以它知道「中国的首都是」后面大概率接「北京」。

但「预测下一个 token」和「给出一个有用、诚实、无害的回答」是两码事。互联网上既有高质量回答,也有胡说八道、有毒、答非所问的内容。模型把它们一起学了。它没有任何内在的「好坏」概念——它只是在模仿语料的分布。

类比

预训练 = 一个读遍了全世界所有书(包括烂书)的人,你问什么他都能接着往下编,而且编得很像那么回事。但他没有「品味」——分不清自己接的这段是金句还是废话。对齐(alignment)就是给这个人装上品味。

1.2对齐三段式:SFT → RM → RL

把一个「只会接话」的模型变成「ChatGPT 那样听话」的模型,经典做法叫 RLHF(Reinforcement Learning from Human Feedback,基于人类反馈的强化学习),由 Christiano 等人(2017)提出框架、OpenAI 的 InstructGPT(Ouyang et al., 2022)发扬光大。它分三步:

①  SFT(监督微调)
    人写「好答案」示范 → 模型模仿
    "问:解释一下黑洞。 答:黑洞是……"  ←人工撰写的高质量回答

②  RM(训练奖励模型)        ← 本系列的主角
    人对模型的多个回答排序 → 训一个模型去模仿这个偏好
    给 (问题, 回答) 打一个分数:越像人喜欢的越高

③  RL(用奖励模型做强化学习)
    让 LLM 自己生成回答 → RM 打分 → 用 PPO/GRPO 等算法
    把「拿高分」的行为强化进 LLM 的参数里
RLHF 三段式。第②步训练的就是「奖励模型」。
名词解释:奖励模型 (Reward Model, RM)

一个输入「(问题, 回答)」、输出一个标量分数的模型。分数越高代表这个回答越符合人类偏好。它本质上是一个回归/打分模型,通常用一个预训练 LLM 当主干、在最后接一个输出单个数字的「头」。

为什么需要单独训一个 RM,而不是直接让人来打分?因为 RL 阶段模型要生成几百万次回答,每次都要一个分数。人打分太慢太贵。于是先用「人排序」的数据训一个 RM,之后让 RM 当「人类品味的廉价自动化替身」,在 RL 里被调用千万次。

1.3奖励模型是怎么训出来的

最经典的方式不是让人直接打绝对分数(「这个答案 7.3 分」——人对绝对分数很不一致),而是让人做两两比较:「A 和 B 哪个更好?」然后用 Bradley-Terry 模型 把「偏好对」变成可训练的损失。

直觉版本:如果人觉得 A 比 B 好,我们就希望 $r(A) > r(B)$。损失函数鼓励「被偏好的那个,分数比另一个高」:

$\mathcal{L} = -\log \sigma\big(r(A) - r(B)\big)$

其中 $\sigma$ 是 sigmoid 函数。当 $r(A)$ 远高于 $r(B)$ 时损失趋近 0;当排反了损失很大。这就是绝大多数 RLHF 奖励模型的训练方式。

深水区:为什么这跟我们项目的 loss 不一样

上面是排序式(pairwise)训练,适合「人类偏好」这种没有客观对错、只有相对好坏的场景。而我们这个项目的 step 有一个 $[0,1]$ 的连续目标分数(某一步「将来能成功的概率」),所以用的是回归式的 masked MSE,而不是 Bradley-Terry。这个区别在 第 5 篇 展开。先记住:奖励模型不止一种训法,训法取决于你的标签长什么样。

1.4结果奖励 ORM:只看终点的问题

上面描述的 RM 给「整个回答」打一个分。在推理任务(数学、代码)里,它通常变成一个更具体的东西:ORM(Outcome Reward Model,结果奖励模型)——只判断最终答案对不对,给整条推理链一个分。

ORM 有个致命的结构性缺陷,叫 稀疏奖励 (sparse reward) 和 信用分配 (credit assignment) 问题:

类比:做错的数学题

一道 20 步的数学题,学生第 14 步算错了符号,导致最终答案错。ORM 只会说:「这条链,0 分。」

但前 13 步可能完全正确!ORM 没法告诉你错在哪一步。反过来,一个学生蒙对了答案但中间步骤全是瞎写,ORM 会给满分。「最终对错」这个信号太稀疏、太粗,没法定位过程里的好坏。

这个缺陷直接催生了下一篇的主角:过程奖励模型(PRM)——不只看终点,而是给每一步都打分。这正是我们整个项目要做的事。

1.5另一条路:RLVR——有些任务根本不用训 RM

在继续之前,必须讲一个对「代码」这个领域极其重要的概念,否则第 3 篇那个核心论点会接不上。

前面说 RM 是「人类品味的自动替身」。但有些任务的好坏不需要人类品味,有客观标准:

  • 数学:答案等于标准答案吗?(可以直接比对)
  • 代码:单元测试通过吗?(直接跑测试)

这类任务,你可以跳过「训一个 RM」这一步,直接用一个规则 / 验证器当奖励:测试过了给 1,没过给 0。这就是 RLVR(Reinforcement Learning with Verifiable Rewards,可验证奖励强化学习),被 DeepSeek-R1、Tülu 3 等近期工作大量使用。

关键直觉:代码是「可验证奖励」的天然乐土

给散文打分需要品味(要训 RM);给代码打分只需要跑测试(规则即奖励)。这就是为什么数学和代码成了近两年 RL 推理研究的主战场——奖励信号又干净又免费。记住这一点,它是第 3 篇论证「为什么代码 Agent 适合做过程监督」的地基。

面试视角

能清楚区分 ORM / PRM / RLVR 三者,是 Reward 方向的入门门槛。一句话版本:ORM 给整条链一个学出来的分;PRM 给每一步一个分;RLVR 干脆不学,用可验证的规则当奖励。 三者不是互斥的——比如你可以用 RLVR 的「测试结果」当 outcome,再用它去推导每一步的 PRM 标签(这正是我们项目干的事,见第 2 篇)。

1.6本篇小结

  • 预训练只教会模型「接话」,不教好坏;对齐补上品味。
  • 经典对齐 = SFT → 训 RM → RL 三段式。奖励模型是 RL 阶段那个被调用千万次的「廉价人类替身」。
  • 奖励模型常用两两偏好 + Bradley-Terry 训练(排序式);但标签是连续分数时改用回归式 loss。
  • ORM 只看终点,带来稀疏奖励 / 信用分配难题 → 催生 PRM。
  • RLVR 对数学/代码这类可验证任务,用规则当奖励,免训 RM。代码是可验证奖励的天然乐土。
