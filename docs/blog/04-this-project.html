第 4 篇 · 工程落地

这个项目的 Phase 1

前三篇是「领域」,这一篇是「你的代码」。把抽象概念逐一对应到仓库里真实的文件、字段和检查。读完你应该能对着任何一行代码说出「它在整条数据流里干嘛、对应哪个学术概念」。

4.1Phase 1 的目标:造一份「带步骤标签」的数据集

整个项目分三阶段:Phase 1 造数据 → Phase 2 训 PRM → Phase 3 评测(Best-of-N)。Phase 1 唯一的产出物是一份数据集:每条代码 Agent 轨迹,每一步都带一个 $[0,1]$ 的质量分。后面两阶段都靠它。

4.2数据流全景

HuggingFace 任务集 (BigCodeBench-Hard)
   │
   ▼  collect_batch.py（异步并发驱动)
   │   给每个 rollout 设环境变量:LOG_DIR / TASK_JSON / ROLLOUT_ID …
   │
   ▼  子进程: node pi/dist/cli.js -p ""     ← TS 代码 Agent 真正干活
   │     └─ pi 加载 extension: trajectory_logger.ts
   │          ├─ 监听 agent 每一步,记录 thought / tool / tool_args / tool_result
   │          ├─ agent_end 时跑 BigCodeBench grader → 得 outcome (0/1) + test_result
   │          └─ 写一行 JSON 到 $LOG_DIR/_.jsonl
   │
   ▼  raw jsonl（含真实 outcome,但 grader 修好前不可信)
   │
   ▼  label_all.py → step_labeler.py
   │     ├─ outcome=0 → 每个 tool step 标 0(simplification,不调 API)
   │     └─ outcome=1 → 每个 tool step 调 K 次 judge(opus→V4-Pro)→ successes/K
   │     └─ 盖章 label_method,原子写 labeled jsonl + manifest
   │
   ▼  30_assemble_dataset.py
         ├─ 跑 6 个 hard check(见 §4.7)
         └─ 切 train/val/test → data/code-trajectory-2.4k/
Phase 1 数据流。TS 端负责「采集 + 判 outcome」,Python 端负责「打步骤标签 + 组装 + 质检」。

4.3为什么用一个真实的 TS Agent(pi)

很多人造 PRM 数据会直接用 API 让模型「一口气写出 CoT」。但本项目要的是真实的多轮 Agent 行为——会读文件、会跑测试、会根据报错改代码。所以它复用了一个开源 TS 编码 Agent pi(yuzheng310/pi,fork 自 earendil-works/pi),通过它的 extension hook 机制把每一步录下来。

名词解释:extension / hook

pi 允许你挂一个扩展脚本,在 Agent 运行的关键时刻(每步结束、整个任务结束)被回调。trajectory_logger.ts 就是这样一个扩展:它不侵入 Agent 的核心逻辑,只是「旁路」监听并记录。靠环境变量 CODE_PRM_LOG_DIR 激活,没设就完全惰性(inert)。

4.4数据契约:三端共享的 schema

TS 写、Python 读、训练器再读——三方必须对同一个数据结构达成一致。这份「契约」定义在 src/labeler/trajectory_schema.py(用 Pydantic),TS 端 trajectory_logger.ts 顶部镜像了同样的 interface。核心结构:

class Step:                  # 一步 = 一次工具调用 或 一个纯思考
    step:        int          # 第几步(0 起)
    thought:     str          # 工具调用前的想法
    tool:        str | None   # 工具名;None=纯思考步
    tool_args:   dict         # 传给工具的参数
    tool_result: str          # 工具返回(上游截断到 8000 字符)
    step_label:  float | None # [0,1] 质量分;labeler 填之前是 None

class Trajectory:
    task_id, task_type, rollout_id
    task_prompt:  str | None  # 题面!judge 没有它就不知道在解什么题
    trajectory:   list[Step]
    outcome:      int         # 0=测试挂  1=测试过
    test_result:  TestResult | None # 比 outcome 更细:exit_code/stderr…
    token_usage:  TokenUsage | None # 真实 API 用量,用来算真实成本
    label_method: "llm_judge" | "outcome_zero_simplification"
                | "mc_rollout" | "ground_truth" | None
设计亮点:label_method —— 诚实命名

每条轨迹都必须盖一个 label_method 章,说明这条数据的标签到底是怎么来的。这是项目里非常成熟的一个决策:它防止下游报告把「LLM-judge 估计」吹成「MC labels」。在 Reward 这种最讲究数据来源的方向,「我对标签来源做了显式可审计的标注」本身就是加分项。 schema 里还有个校验器强制 outcome 必须和 test_result.passed 一致,防止两个字段对不上悄悄污染数据。

4.5outcome 从哪来:BigCodeBench grader

名词解释:BigCodeBench-Hard

一个代码生成基准(Zhuo et al., 2024)。任务给一段需求 + 函数签名,要你实现指定的 entry_point 函数,然后用官方单元测试判对错。Hard 子集更难。注意:v0.1.4 的 Hard 子集只有 148 个任务——这是个真实约束,直接影响数据规模(见第 6 篇)。

要得到第 2 篇所需的「可验证 outcome」,得真的跑测试。这是 grader 的活,也是目前项目已知最危险的一个坑:

坑:test 字段不是能直接跑的脚本

BigCodeBench-Hard 的 test 字段通常只定义了一个 unittest.TestCase 类,既没有 from task import ...(把解法导进来),也没有 unittest.main()(触发运行)。如果 grader 直接把它写进文件 python xxx.py,结果是:0 个测试被执行,但进程 exit 0 → 错误的解法被误判成 outcome=1(假阳性)。

修法:grader 生成临时文件时要 prepend from task import <entry_point> as task_func、append if __name__=="__main__": unittest.main(),这样才会真正执行断言。这是「全量采集前必须修」的头号任务。

坑:并发要 cwd 隔离

全量采集会开 --concurrency 4 多个 rollout 并行。如果它们共享同一个工作目录,A 写的 task.py 会被 B 覆盖,样本互相串台。每个 rollout 必须有独立的 cwd——agent 写的 task.py、grader 文件、__pycache__ 都要落在各自目录里。

4.6step_labeler:把第 2 篇变成代码

这是 PRM 数据的灵魂。对照 src/labeler/step_labeler.py:

def llm_judge_score_step(traj, step_idx, client, K=4):
    prefix = traj.trajectory[: step_idx + 1]      # 取到第 i 步为止的前缀
    prompt = build_continuation_prompt(prefix, …)  # 题面 + 前缀轨迹
    successes = 0
    for _ in range(K):                          # 跑 K 次 judge
        text = client.complete(prompt, temperature=0.9)
        if parses_as_PASS(text):               # judge 说会 PASS?
            successes += 1
    return successes / K                        # soft label ∈ {0/K,…,K/K}

把它和第 2 篇对齐:

  • prefix = trajectory[:i+1] —— 第 2 篇说的「把状态固定在第 i 步」。
  • 跑 K 次、数 PASS 比例 —— Math-Shepherd 的 Soft Estimation,只是用 judge 预测代替真实 rollout。
  • temperature=0.9 —— 让 K 次有随机性,否则 K 次结果一样、比例无意义。
  • parses_as_PASS 用行锚定正则(^\s*OUTCOME:\s*PASS),避免把 "not OUTCOME: PASS" 这种叙述误判成 PASS——一个真实的解析鲁棒性细节。

失败轨迹走 label_trajectory_simplified:直接把每个 tool step 标 0(第 2 篇的 outcome_zero 简化)。整个 label_file 用 原子写(写 .tmp 再 rename),任何中途异常都不会留下半截污染文件。

4.7组装时的 6 个 hard check(质量闸门)

scripts/30_assemble_dataset.py 在切分数据集前会跑 6 个硬检查,任一不过就 SystemExit(2)。这是 Phase 1 的「出厂质检」,每个 check 防一类具体 bug:

检查它防什么
label_method 每条都设了防止有轨迹漏盖标签来源章 → 下游分不清这条怎么标的
outcome=1 的 task_prompt 覆盖率 ≥ 95%judge 没题面就不知道在解什么题 → 标签全废
outcome=1 的 tool step 标签覆盖率 ≥ 80%防止大量步骤没被打分(API 中断/解析失败)
outcome_zero 轨迹的每个 tool step 都 = 0防止「盖了 simplification 章但实际没赋值」的半成品
token_usage 覆盖率 ≥ 80%保证成本统计基于真实用量而非估算
judge 标签均值落在 [0.2, 0.8]防止标签退化(全 0 或全 1 = judge 没在干活,数据没信息量)
面试视角:这 6 个 check 是「数据质量意识」的证据

对 Reward 方向,「你怎么保证训练数据没烂」是核心问题。你可以直接讲:「我在数据组装前设了 6 个 fail-fast 的质量闸门,特别是标签分布非退化检查——如果 judge 退化成全标 1,这批数据等于没信息,我宁可让流水线崩掉也不让脏数据进训练集。」这比「我写了个采集脚本」高了一个量级。

4.8本篇小结

  • Phase 1 产出一份「每步带 $[0,1]$ 标签」的代码 Agent 轨迹数据集。
  • 用真实 TS Agent(pi)+ 旁路 extension 采集真实多轮行为;trajectory_schema.py 是三端共享契约。
  • label_method 诚实命名 + outcome↔test_result 一致性校验 = 数据可审计性。
  • outcome 靠 BigCodeBench grader 跑测试得到;grader harness 假阳性 和 cwd 隔离 是两个必修的坑。
  • step_labeler.py 是第 2 篇 Soft Estimation 的代码化(K 次 judge / successes 比例)。
  • 组装前 6 个 hard check 当质量闸门,尤其「分布非退化」检查体现数据质量意识。
