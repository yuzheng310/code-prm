SERIES OVERVIEW

给代码 Agent 训一个
过程奖励模型

这是一份写给「完全没有 ML / RL 经验」的人的技术博客。读完它,你应该能从零理解什么是奖励模型、什么是过程奖励、为什么代码 Agent 是这件事的好场景,以及这个项目的 Phase 1 到底做了什么、哪里值得讲、哪里是坑。

阅读路径:建议按 0 → 6 顺序读。每篇 10–20 分钟。术语首次出现都会解释。

一句话项目定义

为代码 Agent 的多轮 trajectory(轨迹)训练一个 step-level 的过程奖励模型(Process Reward Model, PRM) —— 让模型不仅能判断「最后做对没做对」,还能判断「中间每一步走得好不好」。

§为什么要读这份东西

如果你只看项目代码,你会看到一堆 Python、一个 TypeScript extension、几个 shell 脚本,然后得出结论:「这不就是个数据采集管道吗?有什么技术含量?」

这个判断对了一半。Phase 1 的代码确实大多是工程胶水。但 Phase 1 背后的问题——「怎么给一个长达几十步的 Agent 轨迹的每一步打一个可信的质量分」——是当前大模型对齐(alignment)和强化学习领域真正没解决好的硬问题之一。深度不在管道里,在那些为了省事而被绕过的设计决策里。

这份博客就是要把那些「被绕过的东西」一个个讲清楚:它们是什么、学术界怎么做的、你这里怎么简化的、简化的代价是什么。

§读完你能回答这些问题

  • 大模型为什么需要「奖励模型」?它和预训练、SFT、RLHF 是什么关系?
  • 「结果奖励 (ORM)」和「过程奖励 (PRM)」差在哪?为什么后者更难也更香?
  • 没有人工标注,怎么自动给每一步打分?(Math-Shepherd 的蒙特卡洛 rollout 思想)
  • 为什么 DeepSeek-R1 公开说「我们放弃了 PRM」,而这个项目还要做 PRM?——这个反问的答案,是整个项目最锋利的论点。
  • 什么是 LoRA、scalar head、masked MSE、Best-of-N、pass@k、step-level F1?
  • 这个项目用 LLM-as-judge 替代真 rollout,踩在了哪条学术趋势上(GenRM)?代价是什么?怎么验证它靠不靠谱?
